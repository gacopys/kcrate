**Step 3: Add `rewriteCreateIndex()` private static method:**

```java
    /**
     * Rewrites CREATE INDEX statements for CrateDB compatibility.
     *
     * PRXY-09: Strip PostgreSQL cast expressions (::type) from index column expressions.
     *   Example: CREATE INDEX idx ON table (col::varchar(250))
     *   Becomes: CREATE INDEX idx ON table (col)
     *   Strategy: regex on the serialized column expression string.
     *   Pattern: \s*::\w+(\(\d+\))? — matches ::varchar, ::varchar(250), ::text, etc.
     *
     * PRXY-10: Strip WHERE clause from partial indexes.
     *   Example: CREATE INDEX idx ON table (col) WHERE col != 'external'
     *   Becomes: CREATE INDEX idx ON table (col)
     *   Strategy: ci.setWhere(null) via JSQLParser before serializing.
     *   Confirmed in Keycloak 26.5.0 OFFLINE_CLIENT_SESSION changesets (2 partial indexes).
     *
     * Also strips the UNIQUE keyword from CREATE UNIQUE INDEX as a safe default (A3):
     *   CrateDB 6.2 behavior with CREATE UNIQUE INDEX is unverified — strip UNIQUE to be safe.
     *   (Per RESEARCH.md Assumption A3 and Pitfall 6)
     *
     * Returns null if the entire index statement should be swallowed (not expected normally).
     */
    private static String rewriteCreateIndex(CreateIndex ci, String originalSql) {
        boolean modified = false;

        // PRXY-10: Strip partial index WHERE clause
        if (ci.getWhere() != null) {
            ci.setWhere(null);
            modified = true;
            System.err.println("[CRATE PROXY] STRIP PARTIAL INDEX WHERE: " + originalSql);
        }

        // Serialize the index to a string so we can apply regex for ::type cast stripping
        String serialized = modified ? ci.toString() : originalSql;

        // PRXY-09: Strip ::type cast expressions from column expressions
        // Regex: \s*::\w+(\(\d+\))? matches ::varchar, ::varchar(250), ::text, ::integer, etc.
        // Applied to the full serialized CREATE INDEX string — cast expressions appear in column lists
        String stripped = serialized.replaceAll("\\s*::\\w+(\\(\\d+\\))?", "");
        if (!stripped.equals(serialized)) {
            serialized = stripped;
            modified = true;
            System.err.println("[CRATE PROXY] STRIP CAST EXPR in CREATE INDEX: " + originalSql);
        }

        // Strip UNIQUE keyword from CREATE UNIQUE INDEX (A3: CrateDB behavior unverified)
        // JSQLParser serializes as "CREATE UNIQUE INDEX ..." — strip the UNIQUE keyword
        if (serialized.startsWith("CREATE UNIQUE INDEX") || serialized.startsWith("create unique index")) {
            // Replace first occurrence of UNIQUE (case-sensitive, at the expected position)
            serialized = serialized.replaceFirst("(?i)\\bUNIQUE\\s+", "");
            modified = true;
            System.err.println("[CRATE PROXY] STRIP UNIQUE from CREATE INDEX: " + originalSql);
        }

        if (!modified) return originalSql;

        logRewrite(originalSql, serialized);
        return serialized;
    }
```

        if (!hasReplicas) {
            List<String> opts = existingOpts != null ? new ArrayList<>(existingOpts) : new ArrayList<>();
            opts.add("(number_of_replicas = '1')");
            ct.setTableOptionsStrings(opts);

<threat_model>
  <threat severity="low">
    <description>The regex `\s*::\w+(\(\d+\))?` used to strip PostgreSQL cast expressions (PRXY-09) is applied to the full serialized CREATE INDEX string. A column name or comment containing `::` followed by a word could be incorrectly stripped — for example a string literal `'foo::bar'` within an index expression.</description>
    <mitigation>Keycloak Liquibase changesets use only simple column expressions in CREATE INDEX — no string literals within index column lists. The regex is safe for the known Keycloak SQL surface. If false positives surface in Phase 3 logs, switch to a JSQLParser AST-level approach for the specific cast expression nodes.</mitigation>
  </threat>
  <threat severity="low">
    <description>The WITH clause injection (PRXY-11) uses a string fallback (`result + " WITH (number_of_replicas = '1')")` if the JSQLParser API does not serialize the WITH keyword. If JSQLParser partially serializes the WITH clause (e.g., emits `WITH` but with incorrect formatting), the fallback may produce malformed SQL like `... WITH (number_of_replicas = '1') WITH (number_of_replicas = '1')`.</description>
    <mitigation>The idempotency check in the test suite (PRXY-11 idempotent test case) catches duplicate WITH injection. Additionally, the guard `if (!withResult.toUpperCase().contains("NUMBER_OF_REPLICAS"))` in the fallback block prevents double-append when JSQLParser did serialize it correctly. Phase 3 migration logs will reveal CrateDB parse errors immediately if the WITH syntax is malformed.</mitigation>
  </threat>
  <threat severity="low">
    <description>Stripping the UNIQUE keyword from CREATE UNIQUE INDEX (A3 mitigation) means CrateDB will receive plain CREATE INDEX instead. This is safe but silently loses uniqueness guarantees that Keycloak may depend on for data integrity.</description>
    <mitigation>Accepted for PoC scope — CrateDB does not enforce referential integrity regardless. Keycloak enforces uniqueness at the application level (e.g., email uniqueness) not purely via DB constraints. If CrateDB 6.2 is later found to support CREATE UNIQUE INDEX, the stripping can be removed.</mitigation>
  </threat>
</threat_model>
